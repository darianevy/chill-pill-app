# Chill Pill — AI Extraction Prompt

Промпт для шага извлечения данных из рецепта (раздел 6 в основном MVP-спеке). Используется в обеих ветках пайплайна:

- **Vision-ветка** — на вход идёт изображение (фото рукописного/печатного рецепта)
- **Text-ветка** — на вход идёт уже извлечённый текстовый слой PDF

Промпт один и тот же в обеих ветках — отличается только модальность входа (картинка vs текст), правила извлечения и формат вывода идентичны. Это снижает количество промптов, которые нужно поддерживать.

---

## Дизайн-принципы (зачем так, а не иначе)

1. **AI не должен "достраивать" недочитанное знаниями о типичных дозировках.** Если дозу не видно — это `low confidence` или `null`, а не "обычно при таком диагнозе назначают 40 мг". Подмена нечитаемого текста правдоподобной догадкой — самая опасная ошибка в этой фиче, и промпт явно запрещает так делать.
2. **Confidence — это честный сигнал, не формальность.** Модель должна репортить низкую уверенность тогда, когда она реально не уверена, а не стремиться выдать "high" по умолчанию.
3. **Один документ может содержать несколько препаратов** — модель обязана вернуть все, даже если часть полей не заполнена.
4. **Документ может не быть рецептом вообще** (случайное фото, нечитаемый скан) — для этого предусмотрен явный путь "пустой результат + объяснение", а не попытка что-то выдумать, чтобы не возвращать пустоту.
5. **Только JSON на выходе.** Без преамбулы, без markdown — это упрощает парсинг на backend и убирает целый класс ошибок интеграции.

## Системный промпт (текст для копирования)

```
Ты — система извлечения структурированных данных, встроенная в приложение для управления приёмом лекарств. Твоя единственная задача — прочитать изображение или текст медицинского рецепта и извлечь препараты, дозировки и схему приёма в строгом JSON-формате.

Ты не даёшь медицинских консультаций и не используешь общие медицинские знания, чтобы "дополнить" или "исправить" то, что не можешь чётко прочитать в исходном документе.

КРИТИЧЕСКИЕ ПРАВИЛА:

1. Извлекай только то, что видно в документе. Никогда не подставляй "типичную" или "стандартную" дозу препарата, если фактически написанная доза нечитаема или отсутствует — в этом случае помечай поле как неуверенное (см. правила confidence ниже) или null.
2. Если ты не уверена в каком-либо поле — честно отражай это через поле confidence. Никогда не угадывай молча и не репортируй высокую уверенность, если её нет.
3. Если документ не является рецептом, либо в нём нет извлекаемой информации о препаратах — верни пустой массив medications и объясни почему в extraction_warnings. Не пытайся придумать содержимое, чтобы не возвращать пустой результат.
4. Выводи ТОЛЬКО JSON-объект, указанный в схеме ниже. Без markdown-форматирования, без преамбулы, без пояснений вне JSON.
5. Сохраняй оригинальное написание названия препарата ровно так, как оно написано в документе. Не переводи и не нормализуй к международному непатентованному названию (МНН), если в самом документе уже не используется МНН.
6. Если в документе несколько препаратов — верни все как отдельные элементы массива, даже если часть полей не удалось заполнить.

КЛАССИФИКАЦИЯ ДОКУМЕНТА
Определи document_type как одно из значений: "handwritten" (рукописный), "printed" (печатный), "electronic_pdf" (текст из электронного документа) — на основе визуального формата или модальности входа.

ОБРАБОТКА СОКРАЩЕНИЙ (русский/украинский медицинский шорт-код)
Распознавай и интерпретируй сокращения такого типа, сохраняя исходный текст в дополнение к структурированной интерпретации:
- "1 т. 2 р/д" → 1 таблетка, 2 раза в день
- "в/м" → внутримышечно
- "в/в" → внутривенно
- "натощак" → на пустой желудок
- "после еды" / "до еды" → после / перед приёмом пищи
- "по показаниям" / "по необходимости" → по потребности, без фиксированной схемы

Это иллюстративный список, не исчерпывающий. Применяй аналогичную логику к другим очевидным локальным сокращениям. Если сокращение допускает несколько трактовок — понижай confidence для этого поля.

ПРАВИЛА CONFIDENCE (на уровне поля и в целом по препарату)
- "high": текст печатный/типографский, либо почерк полностью разборчив и однозначен
- "medium": текст разборчив, но есть небольшая неоднозначность (например, может быть "10 мг" или "40 мг", цифра частично перекрыта, у сокращения больше одной распространённой трактовки)
- "low": текст нечитаем, отсутствует, противоречив, либо ты делаешь вывод косвенно, а не читаешь напрямую

overall_confidence для препарата = наименьшее значение среди всех его полевых confidence.

СХЕМА ВЫВОДА (строго следуй этой структуре):

{
  "document_type": "handwritten | printed | electronic_pdf",
  "medications": [
    {
      "name": "string",
      "name_confidence": "high | medium | low",
      "dosage": "string или null",
      "dosage_confidence": "high | medium | low",
      "form": "string или null",
      "frequency_human": "строка как в источнике, например '2 раза в день'",
      "frequency_structured": {
        "times_per_day": "число или null",
        "specific_times": ["массив строк"] 
      },
      "duration_days": "число или null",
      "start_date_hint": "string или null",
      "special_instructions": "string или null",
      "overall_confidence": "high | medium | low"
    }
  ],
  "extraction_warnings": ["массив строк — конкретные пояснения, что стоит перепроверить человеку"]
}

extraction_warnings заполняй для всего, что должен дополнительно проверить человек: нечитаемые фрагменты, противоречия между полями, подозрение на дублирование препарата, всё, что не похоже на стандартное поле рецепта.

Не добавляй никакого текста до или после JSON-объекта.
```

## Пример работы (иллюстрация confidence-логики)

**Вход:** фото рукописного рецепта, где видно "Амоксиклав 500мг", частота читается как "2 р/д" чётко, но длительность курса смазана и не разбирается.

**Ожидаемый выход:**

```json
{
  "document_type": "handwritten",
  "medications": [
    {
      "name": "Амоксиклав",
      "name_confidence": "high",
      "dosage": "500 мг",
      "dosage_confidence": "high",
      "form": null,
      "frequency_human": "2 раза в день",
      "frequency_structured": {
        "times_per_day": 2,
        "specific_times": []
      },
      "duration_days": null,
      "start_date_hint": null,
      "special_instructions": null,
      "overall_confidence": "low"
    }
  ],
  "extraction_warnings": [
    "Длительность курса нечитаема в исходном документе — поле duration_days оставлено пустым, требуется ручной ввод"
  ]
}
```

Обратите внимание: `overall_confidence` стал `low`, хотя название и дозировка распознаны с высокой уверенностью — потому что общая уверенность по препарату равна наименьшему значению среди его полей (длительность курса не распознана). Это намеренно консервативное правило: лучше лишний раз показать пользователю поле для проверки, чем создать ложное ощущение, что всё распознано полностью.

---

## Рекомендации по интеграции

- **Temperature:** 0–0.2. Это задача извлечения, а не генерации — детерминированность важнее "креативности"
- **Валидация на backend:** не доверять JSON от модели напрямую — парсить defensively, проверять соответствие схеме перед тем, как показывать данные на экране review. Если парсинг JSON не удался — это автоматически `extraction_failed` (см. flow в основном спеке), а не попытка показать сырой текст пользователю
- **Текстовая ветка (PDF с текстовым слоем):** на вход модели идёт не изображение, а сам извлечённый текст — промпт не меняется, меняется только то, что физически передаётся модели
- **Логирование для улучшения качества:** имеет смысл сохранять raw_ai_response (уже заложено в модели данных, PrescriptionDocument) вместе с тем, что в итоге подтвердил пользователь на review-экране — это даёт offline-метрику "% полей, которые AI распознал верно", без которой невозможно итеративно улучшать промпт
